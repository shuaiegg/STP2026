# STP2026 全站上线前审计计划 (Pre-Launch Audit Masterplan)

**审计目标**：确保 STP 从“本地开发项目”跃升为“可商业化变现的 SaaS 产品”，达到零死链、零错版、文案极具煽动性的工业级标准。
**审计范围**：Public Frontend (门面), Protected Dashboard (用户端), Admin Panel (管理端)。
**审计维度**：文案 (Copywriting), 设计 (UI/UX), 页面逻辑 (Logic & Routing)。

---

## 阶段一：前端门面大盘查 (Public Frontend)
**目标人群**：未注册的潜在客户。
**核心诉求**：建立信任，讲清价值，促成注册。

### 1. 首页 (`/`)
*   **文案审计 (Copywriting)**：
    *   H1 标题是否能在 3 秒内讲清楚“我们是谁，解决什么痛点”？
    *   是否有过多技术自嗨（如满屏的 RAG、LLM），而缺乏商业利益点（如 ROI、流量增长）？
*   **设计审计 (UI/UX)**：
    *   首屏是否有强力的视觉锚点（Hero Image/Video）？
    *   移动端首屏是否能完整显示主要的 CTA (Call to Action) 按钮？
*   **逻辑审计 (Logic)**：
    *   所有 CTA 按钮（如“立即试用”、“了解更多”）是否都指向正确的 `/register` 或 `/pricing`？

### 2. 定价与转化 (`/pricing`, `/register`, `/login`)
*   **文案审计**：
    *   定价方案的区分度是否明显（例如：Starter 解决单人痛点，Pro 解决团队效率）？
    *   注册/登录页面的提示语是否温和且专业，有无错别字（如“登入”vs“登录”的统一）？
*   **设计审计**：
    *   高利润的套餐是否有高亮/推荐标签（"Most Popular"）？
    *   表单输入框的错误提示（如密码过短）是否为红色且清晰？
*   **逻辑审计**：
    *   Oauth（Google/GitHub 登录）配置是否正确，回调 URL 是否有效？
    *   注册成功后，是否能顺滑跳转至 `/dashboard`？

### 3. 博客与内容 (`/blog`, `/course`)
*   **审计重点**：
    *   文章列表的缩略图比例是否统一？
    *   点击任意文章是否能正常进入 SSR（服务器渲染）详情页，有无 404？

## 阶段二：用户后台深潜 (Protected Dashboard)
**目标人群**：已付费或试用中的用户。
**核心诉求**：工具好用，数据清晰，没有卡顿。

### 1. 核心工作台 (`/dashboard`, `/tools/geo-writer`)
*   **文案审计**：
    *   功能入口的描述是否能让“小白用户”一目了然？（如不用“触发智能体”，而用“开始创作”）。
    *   加载状态的文案是否能缓解焦虑（如“正在查阅全球 100 篇竞品文章...”）？
*   **设计审计**：
    *   侧边栏 (Sidebar) 和主内容区的留白是否合理，是否显得拥挤？
    *   夜间模式（如果有）或强光下的对比度是否达标？
*   **逻辑审计**：
    *   路由守卫 (Route Guard)：清空本地 Cookie 后，强制访问该路径是否会被立刻踢回 `/login`？
    *   工具使用过程中（如我们刚测完的 geo-writer），中途刷新页面，状态是否会优雅丢失并给出提示？

### 2. 资产与账户 (`/dashboard/library`, `/dashboard/billing`)
*   **逻辑与设计审计**：
    *   历史文章列表的分页或滚动加载是否正常？
    *   积分/额度 (Credits) 的数字是否突出？当积分不足时，是否有醒目的“充值”按钮？

---

## 阶段三：管理中枢安全排查 (Admin Panel)
**目标人群**：网站所有者 (Jack)。
**核心诉求**：绝对安全，数据总览无死角。

### 1. 权限与边界墙 (`/admin/*`)
*   **逻辑审计 (最高优先级)**：
    *   **越权测试**：新注册一个“纯普通测试账号”（如 `test_user@scaletotop.com`），强行通过 URL 访问 `/admin/users` 等管理端路由，系统必须返回 403 Forbidden 或重定向到 `/dashboard`。
    *   **超管验证**：只有真正具备 `role === 'ADMIN'` 的账号（如 `jack47.chn@gmail.com`）才能看到侧边栏的“管理后台”入口并成功进入。

### 2. 数据与功能操作
*   **文案与逻辑审计**：
    *   是否有“一键封禁用户”或“手动加积分”的操作面板，且点击时有“二次确认 (Are you sure?)”弹窗？
    *   关键数据的表格（如新注册用户数、消耗的 API 额度）是否有溢出屏幕的 UI bug？

---

## 🎯 审计执行方式 (The "How-To")

我们将调动 STP-Dev-Squad 兵分三路：
1.  **文案狗 (PM-Aladdin)**：扫描全站所有 Hardcode 的中文/英文文本，输出 `COPY_REPORT.md`。
2.  **视觉猫 (UX-Aladdin)**：通过 `agent-browser` 截图首页、控制台和手机端，比对设计规范。
3.  **渗透鼠 (QA-Aladdin)**：专门做破坏性测试（没登录瞎点、越权访问 Admin、表单瞎填），寻找逻辑漏洞。
